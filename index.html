0TL;DR — For the Impatient

The binary takes exactly 64 bytes of input, scrambles them through a secret linear transform, and verifies the result against a hardcoded target array. Rather than spending hours reverse-engineering the math by reading raw assembly, we treat the binary as a black box. We probe it 65 times with carefully chosen inputs to completely recover the hidden 64×64 transformation matrix. By computing the matrix inverse modulo 256, we can solve for the exact flag instantly.

2The Key Insight — It's All Linear, Baby

Challenge title: "My Favorite Ingredient"
One ingredient. One repeated operation.
That ingredient? Linearity.
When the challenge literally gives you the methodology in the name 🧂

Before dissecting the assembly, we must answer one critical question: Is the transformation linear? If it is, we can bypass the assembly entirely.

What is a linear transform? (Explain Like I'm 5 Edition 🍼)

Imagine a magic blender. You drop in 64 distinct ingredients (bytes), and it produces 64 newly mixed liquids. The blender is linear if every resulting liquid is simply a fixed ratio of the input ingredients added together. There are no exotic reactions—no squaring, no conditional statements, just pure addition and multiplication by fixed amounts.

Matrix inversion is the mathematical equivalent of un-blending a smoothie using a secret recipe card. If you know the exact percentage of apples and bananas used, and they were purely mixed, you can calculate precisely what went in! Mathematically, this is expressed as: output = M · v + c, where M is a 64×64 matrix representing the "recipe ratios," v is your 64-byte input, and c is a constant offset.

The Binary Execution Pipeline

Our objective: Reverse the arrow. Start with the known target and recover v.

Why Linearity is Your Superpower

A linear map over 64 bytes can be completely reconstructed using just 65 deliberate inputs. The methodology is as follows:

1

Establish the Baseline

Input 64 zero bytes. The resulting output will be M · 0 + c = c. This allows us to capture the constant offset instantly.

2

Probe Individually

For each byte position i: send an input where only byte i is 1 and the rest are 0. The output yields M · eᵢ + c.

3

Extract the Columns

Subtract the baseline (modulo 256) from the output. This calculation extracts exactly column i of matrix M.

4

Matrix Recovered

With 1 baseline run and 64 probes (65 total), the entire 64×64 matrix is ours. No assembly analysis required.

3Step-by-Step Solve

Step 3.1 — Intercept the Working Buffer

The binary only outputs a binary pass/fail state. To reconstruct the matrix, we need visibility into the 64-byte transformed buffer just before the memcmp executes. We can achieve this using GDB to halt execution and dump the memory.

⚠️
Mind Your Registers! (System V AMD64 ABI)

The memcmp function requires two memory pointers. Under the System V AMD64 ABI (the calling convention dictating how functions pass data in Linux), the first argument resides in $rdi, and the second in $rsi. If Ghidra decompiles to memcmp(tmp, expected, 64), our working buffer tmp is located in $rdi! Dumping the wrong register will extract the hardcoded target instead, ruining the matrix recovery.

Action: Create a file named dump.gdb in your working directory and paste the following commands to automate the memory extraction:

dump.gdb 
# Halt execution right at the comparison
break *0x<memcmp_address_from_ghidra>

# Dump 64 bytes from $rdi (the first argument) and continue
commands
silent
dump binary memory /tmp/buf.bin $rdi $rdi+64
continue
end

# Run while piping our binary probe from a file
run < /tmp/probe.bin

Step 3.2 — Extract the Target Array

We need the hardcoded target bytes to solve the equation. Navigate to the .data section in Ghidra, locate the expected array, right-click, select Copy Special -> Python Byte String, and paste it securely into your exploit script.

Step 3.3 — Execute the 65 Probes

Now we implement the probing logic in Python. This script automates running GDB 65 times and extracting the resulting matrix columns.

python — matrix recovery 
import subprocess, os

def run_probe(inp: bytes) -> bytes:
    """Write probe to file, execute via GDB, and read the dumped buffer."""
    with open('/tmp/probe.bin', 'wb') as f:
        f.write(inp)
    subprocess.run(
        ['gdb', '-batch', '-x', 'dump.gdb', './my-favorite-ingredient'],
        capture_output=True
    )
    return open('/tmp/buf.bin', 'rb').read()

# ── 1: Capture the constant offset (Baseline) ──
baseline = run_probe(bytes(64))
print(f"Baseline (c) retrieved.")

# ── 2: Probe the matrix columns ──
matrix_cols = []

for i in range(64):
    probe = bytearray(64)
    probe[i] = 1  # Create unit vector
    output = run_probe(bytes(probe))

    # Modulo 256 subtraction isolates the column data
    col = bytes((b - base) % 256 for b, base in zip(output, baseline))
    matrix_cols.append(col)

print("64x64 Matrix successfully recovered!")
🚫
Crucial: The Modulo 256 Constraint

Python integers have infinite precision. If you subtract bytes without applying % 256, Python returns negative integers instead of correct byte wrap-arounds. This will silently corrupt your matrix and yield a garbage flag.

Step 3.4 — Matrix Inversion & Flag Retrieval

With M, c, and target t in our possession, we apply elementary algebra modulo 256:

M · v + c = t
(subtract c)
M · v = t − c
(multiply by M⁻¹)
v = M⁻¹ · (t − c)

We invert the matrix using Gaussian elimination over the Z/256Z ring.

python — inversion & solve 
def mat_inv_mod256(M):
    """Gaussian elimination modulo 256."""
    n = len(M)
    aug = [[M[i][j] for j in range(n)] + [1 if i==j else 0 for j in range(n)]
            for i in range(n)]

    for col in range(n):
        # Locate an invertible pivot (odd number)
        pivot = None
        for row in range(col, n):
            if aug[row][col] % 2 == 1:
                pivot = row; break
        aug[col], aug[pivot] = aug[pivot], aug[col]

        # Compute modular inverse (Requires Python 3.8+)
        inv_piv = pow(aug[col][col], -1, 256)

        # Eliminate column elements
        aug[col] = [(v * inv_piv) % 256 for v in aug[col]]
        for r in range(n):
            if r != col:
                f = aug[r][col]
                aug[r] = [(aug[r][j] - f * aug[col][j]) % 256
                          for j in range(2*n)]

    return [[aug[i][n+j] % 256 for j in range(n)] for i in range(n)]


# Prepare the Right-Hand Side (RHS)
target = bytes(...)  # Hardcoded array extracted from Ghidra
rhs = bytes((t - b) % 256 for t, b in zip(target, baseline))

# Transpose columns into rows for proper matrix multiplication
M_rows = [[matrix_cols[j][i] for j in range(64)] for i in range(64)]
M_inv = mat_inv_mod256(M_rows)

# Multiply M_inv by RHS to retrieve the flag
flag = bytes(
    sum(M_inv[i][j] * rhs[j] for j in range(64)) % 256
    for i in range(64)
)

print(f"Flag: {flag.decode()}")
🔄
Matrix Orientation is Paramount

The probing phase yields columns, but standard array syntax expects rows. A failure to transpose the matrix will output plausible, printable ASCII characters that ultimately fail the binary check. Always transpose before inversion!

Wisdom

6Key Takeaways

🔬
Verify Linearity First

Testing for linearity using a few chosen inputs takes mere minutes and can save hours of tedious manual assembly analysis. Always check for this property before opening the decompiler.

📏
Fixed Length Indicates Fixed Block Transforms

A strict input length check (like exactly 64 bytes) is a powerful indicator of a fixed-width block transformation—an ideal candidate for matrix recovery techniques.

🌍
The Fundamental Lesson

A verification algorithm that appears cryptographic is entirely distinct from one that is cryptographic. Computational complexity is an illusion if the underlying operation remains linear and deterministic. Black-box dynamic analysis cuts cleanly through any obfuscation layered on top of linear operations.
